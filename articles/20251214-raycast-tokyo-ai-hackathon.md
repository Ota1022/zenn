半年以上が経ったある日、Raycast・Anthropic・Vercel・Supabase がコラボしてハッカソンを開催するという情報を X で見かけました。Connpass でよくテックイベントに参加しているのですが、この豪華な座組には驚きました。加えて、いまいち活用できていない Raycast を使いこなす絶好の機会だと感じ、チームが組めなくてもソロで参加すればいいかと思い、参加登録しました。

### 4.1 イベントの雰囲気

![ハッカソン当日の様子](/images/20251214/2_hackathon_discussion.jpg)

会場は AWS Startup Loft Tokyo。参加者の体感3割以上がグローバルなエンジニアで、イベント全体が英語で進行されるという、かなりグローバルな雰囲気でした。普段参加している Connpass のテックイベントとはまた違った、「1日でプロダクトを作り上げる」ハッカソンならではの緊張感があったのを覚えています。

### 4.2 タイムスケジュールと開発環境

当日のタイムスケジュールは以下のような流れでした。細かい時間は多少前後しましたが、15:00の開場から21:45の終了まで、約7時間の長丁場でした。

![ハッカソンのタイムテーブル](/images/20251214/2_hackathon_timeline.png)

16:30から本格的に開発がスタートし、19:45の提出締め切りまで黙々とコーディングする時間が続きます。開発中は Raycast、Supabase、Vercel の各スタッフが会場を回り、必要に応じて技術的なサポートをしてくれました。

特筆すべきは、参加者全員に事前配布された **Claude API の100ドル分のクレジット**（期限付き）です。これにより、Claude Code を API モードで実質使い放題という環境が整っていました。さらに、このAPIクレジットは単に開発支援ツールとして使うだけでなく、作成する Extension の中でモデルを呼び出す API としても活用できたため、AI 機能をフル活用した Extension 開発が可能でした。

### 4.3 チーム構成と開発体制

当日は、プレイベントで一緒にアイデアを出していたメンバーを中心に、4 人でチームを組みました。

役割分担は以下のような形で進めました：

* **コア機能実装**: クリップボード履歴の取得と AI による手順書生成ロジック
* **UI 実装**: Raycast Extension の画面設計とユーザー体験の調整
* **API 統合**: Claude API との連携と LocalStorage の実装
* **ピッチ準備**: デモシナリオとスライドの作成

開発を進める上で意識したのは以下の点です：

* 1日で動くものを作るために、最初の1時間で「MVPの境界」を決める
* UIは Raycast の標準コンポーネントに寄せて、迷いを減らす
* "それっぽい体験"を早めに作って、途中でピッチの筋を崩さない

コミュニケーションツールとしては Slack でチャンネルを作り、実装の進捗や技術的な相談を共有しながら進めました。タスクは「クリップボード履歴の表示」「メモ機能の追加」「AI による生成」と機能単位で区切り、並行して開発できるよう調整しました。

### 4.4 作成した Raycast Extension「Trayce」

#### どんなものを作ったか

「作業手順をあとから書こうと思っても、どのコマンド打ったか思い出せない」という経験、ありませんか？

Trayce は、**クリップボード履歴から作業手順書を自動生成する** Raycast Extension です。普段コピーしているコマンドや URL に簡単なメモを足すだけで、Claude が Markdown の手順書に整形してくれます。

使い方はシンプルで、作業中にコピーしたものを Raycast で開いて、各項目に「何のためにやったか」をメモして、生成ボタンを押すだけ。あとは Claude がいい感じに Markdown の手順書にしてくれるので、それを Notion や Jira に貼り付けて共有できます。

主な機能：
- クリップボード履歴の取り込みと分類（コマンド/URL/コードなど）
- 各項目へのメモ追加
- Claude API による手順書生成
- 生成した手順のローカル保存

![Trayce のデモ](/images/20251214/2_demo_trayce_full_960.gif)

#### 技術スタックと開発の工夫

構成：Raycast Extension + Anthropic API + Raycast LocalStorage

1日という時間制約の中で、Claude Code（API モード）をフル活用しながら開発を進めました。工夫したのは、最初に MVP の範囲を明確にして「クリップボード履歴→メモ追加→生成」という最小限のフローに絞ったことです。UI も Raycast の標準コンポーネントに寄せて、デザインに悩む時間を削減しました。

コマンドは「手順を作る」と「手順を見返す」の2つだけ、データモデルも必要最小限、状態管理も複雑にせず LocalStorage への読み書きをシンプルに。このあたりの割り切りが、1日で動くものを作る上で重要でした。

なお、クリップボードの内容は生成のために直接 Anthropic API に送られるため、その点は注意が必要です。

:::message
Raycast Extension の作成手順については、別記事「**Raycast Extension 開発のすすめ**」で詳しく解説しています。雛形の作成から Store 公開までの流れを ToDoリストを題材にステップバイステップで説明しているので、ぜひ作ってみてください！
:::

### 4.5 ピッチとファイナリスト発表

19:45 の提出締め切り後、各チームが順番にピッチを行いました。限られた時間の中で、作ったものの価値と技術的な工夫を伝えるのは簡単ではありませんでしたが、デモを交えながら「クリップボード履歴から手順書を作る」というコンセプトを説明しました。

特に印象的だったのは、**英語でピッチを行う**という経験でした。普段のテックイベントでは日本語で発表することがほとんどだったので、短時間で準備したプロダクトを英語で説明するのは新鮮で、とても良い経験になりました。限られた時間の中で、技術的な工夫と価値を端的に伝える難しさを実感しましたが、グローバルなエンジニアが集まる環境だからこその緊張感と充実感がありました。

![ファイナリスト発表の様子](/images/20251214/2_hackathon_final.jpg)

結果として、私たちのチームはファイナリストには選ばれませんでしたが、チームメンバー全員がやり切った感覚があり、大きな達成感を得られました。ファイナリストに選ばれたチームのプレゼンテーションを見て、デモ動画の作り込みやアイデアの切り口の面白さ、そして見せ方の上手さに圧倒されました。特にストーリーテリングの巧みさや、ユーザー体験を視覚的に伝える演出は、まだまだ自分たちには場数が足りないと感じさせられる学びの多いものでした。

すべてのチームのピッチが終わった後、審査員による評価が行われ、ファイナリストが発表されました。会場には様々なアイデアの Extension が集まり、それぞれが異なる角度から「AI × Raycast」の可能性を追求していたのが印象的でした。MCP サーバーを活用した高度な統合から、日常の小さな不便を解消するシンプルなツールまで、多様性に富んだピッチが繰り広げられました。

日本コミュニティの活発さについて非常に感謝している様子で、Extensions の開発やフィードバックがプロダクトの進化に大きく貢献しているとのことでした。特にこのハッカソンのような取り組みが、新しいユースケースの発見につながっていると話していました。

### 5.2 Windows 版 Raycast のデモを見て感じたこと

![Windows 版 Raycast のデモ](/images/20251214/3_meetup_win_demo.jpg)

Meetup では Windows 版 Raycast のデモが披露されました。macOS 版と同様の UI/UX を Windows で実現しており、クロスプラットフォーム対応の完成度の高さに驚きました。

また、Windows 版のリリースによって Raycast のユーザーベースがさらに拡大することで、コミュニティがより活発になり、新しいアイデアや Extensions が生まれる可能性が高まると感じました。自分の開発スタイルとしても、Windows ユーザーを意識した Extension 開発を考えていきたいと思っています。

## 6. イベントを通じて得た学びとこれから

3 日間のイベントを通して、技術的な学び以上に、プロダクト開発やコミュニティへの向き合い方について考えさせられることが多くありました。

### 6.1 アイデアは「新しさ」より「痛みの具体性」で勝負する

プレイベントでブレストしていた時、最初は「AI を使った新しい体験」を追いかけようとしていました。しかし、最終的に採用したアイデアは「クリップボード履歴から手順書を作る」という、地味に聞こえるものでした。

このアイデアが強かったのは、**誰もが経験する具体的な痛み**があったからです。「手順書を書くのは面倒」「でも聞かれたら答えなきゃいけない」「あれ、何やったっけ？」というフラストレーションに、チームメンバー全員が共感できました。

ハッカソンでは、新規性を追求するより、「この痛み、自分も感じてた！」と言ってもらえるアイデアの方が、開発中のモチベーションも維持しやすく、ピッチでも伝わりやすいと実感しました。

### 6.2 1日開発は「体験の導線」を一本通すのが最優先

7時間という制約の中で、機能を詰め込みたい誘惑に何度も駆られました。「履歴のフィルタリング機能も欲しい」「複数の手順を比較できたら」など、アイデアはいくらでも出てきます。

しかし、最終的に完成したのは「履歴を選ぶ→メモを足す→生成する」というシンプルな一本道でした。この判断が正解だったと思います。ピッチで実際にデモを動かした時、観客が「あ、これなら使える」と理解できる体験を届けられたからです。

1日で作るプロダクトは、**ユーザーが迷わず最後まで辿り着ける道を一本作る**ことに集中すべきだと学びました。複数の機能より、一つの完結した体験の方が価値があります。

### 6.3 コミュニティイベントは「自分の現在地を更新する場」

Meetup での交流を通じて、Raycast コミュニティのグローバルな広がりと多様性を実感しました。台湾のメンバーが地域特化の Extension を作っていたり、他のチームが MCP サーバーを使いこなしていたり、自分が知らない活用法がたくさんありました。

こうしたイベントは、単に新しい情報を得るだけでなく、**「自分は今どこにいて、次にどこを目指すか」を再確認する機会**になると感じました。他の参加者の取り組みを見て、自分の Raycast の使い方がまだまだ浅いことに気づき、もっと深く活用したいという意欲が湧きました。

### 6.4 AI 時代の開発スタイル

今回のハッカソンでは、Claude API のクレジットが配布され、開発中に Claude Code を使い放題という環境でした。これにより、Raycast Extension の雛形生成からデバッグまで、かなりの部分を AI に支援してもらいながら開発できました。

特に印象的だったのは、「やりたいこと」を言葉で伝えるだけで、ある程度動くコードが生成される体験です。もちろん細かい調整は必要ですが、0→1 のスピードが圧倒的に速くなりました。

AI を活用することで、**アイデアから動くプロトタイプまでの時間が劇的に短縮される**時代になったと実感しました。これからは、コーディングスキル以上に、「何を作るか」「どんな体験を届けるか」というアイデアとデザインの力が問われると感じています。

### 6.5 これからやりたいこと

今回の経験を通じて、Raycast Extension 開発の面白さに目覚めました。今後取り組んでいきたいことをいくつか挙げておきます：

**Trayce の改善と公開**

ハッカソンで作った Trayce は MVP としては動きますが、まだ改善の余地がたくさんあります。複数の手順の管理機能や、チーム内での共有機能など、実際に使いながらブラッシュアップして、最終的には Raycast Store での公開を目指したいと思っています。

**日常業務で使える Extension の開発**

Raycast の真価は、日常の小さな摩擦を減らすことにあると感じています。自分の業務フローを見直して、「これ、毎日やってるな」という作業を Extension 化していきたいです。例えば、よく使う社内ツールとの連携や、定型的な作業の自動化などです。

**コミュニティへの貢献**

日本の Raycast コミュニティがこれだけ活発なのを見て、自分も何か貢献したいと思うようになりました。作った Extension を公開することはもちろん、イベントでの LT や記事の執筆など、自分の経験を共有していきたいです。

**Windows 版対応の準備**

Windows 版 Raycast のリリースを見据えて、クロスプラットフォームで動く Extension の開発ノウハウを蓄積しておきたいです。Windows ユーザーにも使ってもらえる Extension を作ることで、より多くの人に価値を届けられると思っています。

---

3 日間のイベントを通じて、Raycast というツールの可能性と、それを支えるコミュニティの温かさに触れることができました。「便利なツールを使う」だけでなく、「自分でも拡張できる」という体験は、エンジニアとしてのワクワク感を思い出させてくれました。

次は、自分が作った Extension で誰かの日常を少し楽にできるよう、開発を続けていきたいと思います。
